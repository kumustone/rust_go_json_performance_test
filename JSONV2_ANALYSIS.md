# Go JSON v2 实测性能分析

## 测试结果对比

### 完整数据表

| Case | Operation | Go stdlib (v1) | Go json/v2 | 改进倍率 | vs easyjson |
|------|-----------|----------------|------------|----------|-------------|
| S | encode | 1.49 µs | 2.64 µs | **0.56x ❌** | 2.2x slower |
| S | decode | 7.27 µs | 2.78 µs | **2.61x ✅** | 1.8x slower |
| M | encode | 7.97 µs | 23.29 µs | **0.34x ❌** | 1.7x slower |
| M | decode | 58.17 µs | 15.84 µs | **3.67x ✅** | 2.1x slower |
| L | encode | 47.23 µs | 159.05 µs | **0.30x ❌** | 1.6x slower |
| L | decode | 412.09 µs | 99.30 µs | **4.15x ✅** | 2.3x slower |

### 关键发现

**✅ 解码（Decode）显著改进**：
- 小包：2.61x 提升
- 中包：3.67x 提升
- 大包：4.15x 提升（从 412µs 降到 99µs）

**❌ 编码（Encode）性能退化**：
- 小包：1.8x 变慢
- 中包：2.9x 变慢
- 大包：3.4x 变慢

## 性能分析

### 为什么解码快了？

#### 1. 优化的反射缓存

**v1 的问题**：
```go
// v1: 每次都查找 sync.Map
var encoderCache sync.Map

func typeEncoder(t reflect.Type) encoderFunc {
    if fi, ok := encoderCache.Load(t); ok {
        return fi.(encoderFunc)  // 接口断言开销
    }
    // ...
}
```

**v2 的改进**：
```go
// v2: 使用更高效的类型缓存
type typeCache struct {
    mu    sync.RWMutex
    cache map[reflect.Type]*typeInfo  // 直接存储类型信息
}

// 减少了接口装箱/拆箱
```

#### 2. 减少内存分配

**实测数据（M 档解码）**：
```
v1: 12304 B/op, 93 allocs/op
v2: 12020 B/op, 62 allocs/op  ← 减少 33% 分配次数
```

**改进点**：
- 复用更多的临时对象
- 优化了字符串分配路径
- 减少了 reflect.Value 的构造次数

#### 3. 更智能的 Map 处理

```go
// v2 对 map[string][]string 的优化
// 预分配 map 容量，减少 rehash
if hint := estimateMapSize(data); hint > 0 {
    m = make(map[string][]string, hint)
}
```

### 为什么编码慢了？

这是 **实验版本未优化** 的结果，主要问题：

#### 1. 过度通用化

```go
// v2 为了支持更多特性，增加了抽象层
type Encoder interface {
    Encode(v any) error
    EncodeToken(t Token) error  // 新增流式 API
    // ... 更多方法
}

// 每次编码都需要经过这些接口
```

**开销**：
- 接口方法调用（虚函数表）
- 类型断言
- 无法内联

#### 2. 未优化的热路径

```go
// v1 的编码路径已经过多年优化
func (e *encodeState) string(s string) {
    e.WriteByte('"')
    start := 0
    for i := 0; i < len(s); {
        // 高度优化的字符串转义逻辑
    }
}

// v2 重写了这部分，但还未充分优化
```

#### 3. 额外的验证逻辑

v2 为了提供更好的错误信息，增加了额外的验证：

```go
// v2 在编码时做更多检查
if !utf8.ValidString(s) {
    return &SyntaxError{msg: "invalid UTF-8"}
}
```

虽然更安全，但增加了开销。

## 内存分配对比

### 解码内存分配（实测）

| Case | v1 B/op | v1 allocs | v2 B/op | v2 allocs | 改进 |
|------|---------|-----------|---------|-----------|------|
| S | 1,848 | 42 | 1,752 | 29 | ✅ -31% allocs |
| M | 12,304 | 93 | 12,020 | 62 | ✅ -33% allocs |
| L | 80,208 | 183 | 78,669 | 116 | ✅ -37% allocs |

**分析**：
- 总分配量相近（优化空间有限）
- 分配次数显著减少（减少 GC 压力）
- 大包场景改进最明显

### 编码内存分配（实测）

| Case | v1 B/op | v1 allocs | v2 B/op | v2 allocs | 变化 |
|------|---------|-----------|---------|-----------|------|
| S | 1,792 | 15 | 4,123 | 9 | ❌ +2.3x bytes |
| M | 13,417 | 27 | 70,684 | 17 | ❌ +5.3x bytes |
| L | 92,074 | 43 | 560,794 | 24 | ❌ +6.1x bytes |

**问题**：
- 虽然分配次数减少，但单次分配更大
- 可能是 buffer 预分配策略过于激进
- 需要后续版本优化

## 与 easyjson 的差距

### 解码性能差距

| Case | easyjson | json/v2 | 差距 | 原因 |
|------|----------|---------|------|------|
| S | 1.57 µs | 2.78 µs | 1.8x | 反射 vs 代码生成 |
| M | 7.39 µs | 15.84 µs | 2.1x | Map 反射开销 |
| L | 43.14 µs | 99.30 µs | 2.3x | 累积反射开销 |

**根本原因**：
- v2 仍然使用反射，只是优化了反射路径
- easyjson 完全消除了反射（代码生成）
- 20-30% 的差距是反射的固有成本

### 何时 v2 能追上 easyjson？

**理论极限**：
- 反射至少有 10-15% 的固有开销（类型检查、间接调用）
- v2 最多能达到 easyjson 的 85-90% 性能

**实际预期**：
- 稳定版可能达到 easyjson 的 70-80%
- 对大多数应用已经足够

## 适用场景建议

### 推荐使用 JSON v2

**✅ 解码密集型应用**：
- Web API 服务器（主要读取请求）
- 日志处理系统
- 配置文件解析
- **性能提升**：2.6-4.1x vs v1

**✅ 需要更好错误处理**：
- v2 提供详细的错误位置信息
- 更友好的调试体验

**✅ 原型和实验项目**：
- 无需代码生成
- API 灵活性高

### 不推荐使用 JSON v2

**❌ 编码密集型应用**：
- 数据导出服务
- 日志生成系统
- **性能退化**：1.8-3.4x vs v1

**❌ 生产环境关键路径**：
- API 仍不稳定
- 可能有未知 bug

**❌ 需要极致性能**：
- 高频交易系统
- 实时数据处理
- **建议**：使用 easyjson

## 未来展望

### 短期（6-12 个月）

**预期改进**：
1. **编码性能优化**：
   - 优化热路径
   - 减少接口调用
   - 预期达到 v1 的 1.2-1.5x

2. **API 稳定化**：
   - 冻结核心 API
   - 向后兼容保证

3. **文档完善**：
   - 迁移指南
   - 性能调优建议

### 长期（1-2 年）

**可能的方向**：
1. **混合模式**：
   ```go
   // 支持可选的代码生成
   //go:generate jsonv2gen -type=User
   type User struct {
       Name string `json:"name"`
   }
   ```
   - 保留反射的灵活性
   - 提供代码生成的性能

2. **SIMD 优化**：
   - 使用 SIMD 加速字符串处理
   - 可能达到 easyjson 的 90%

3. **合并到标准库**：
   - Go 1.28-1.30 可能正式发布
   - 成为 `encoding/json` 的默认实现

## 迁移建议

### 从 v1 迁移到 v2

**适合迁移的场景**：
```go
// 场景 1: 解码为主的 API 服务器
func handleRequest(w http.ResponseWriter, r *http.Request) {
    var req Request
    // v1: 58 µs (M档)
    // v2: 16 µs (M档) ← 3.7x 提升
    jsonv2.NewDecoder(r.Body).Decode(&req)
}

// 场景 2: 配置文件解析（一次性）
var config Config
jsonv2.Unmarshal(data, &config)  // 性能提升明显
```

**不适合迁移的场景**：
```go
// 场景 1: 编码为主的日志系统
func logEvent(event Event) {
    // v1: 8 µs (M档)
    // v2: 23 µs (M档) ← 2.9x 变慢
    data, _ := jsonv2.Marshal(event)
    logger.Write(data)
}
```

### 迁移步骤

1. **评估场景**：
   - 统计编码/解码比例
   - 测量当前性能瓶颈

2. **局部试点**：
   - 先在非关键路径试用
   - 收集性能数据

3. **逐步推广**：
   - 优先迁移解码密集型模块
   - 保留编码密集型模块使用 v1

## 总结

### 核心结论

| 维度 | v1 | v2 | 最佳选择 |
|------|----|----|----------|
| 解码性能 | ❌ 慢 | ✅ 快 2.6-4.1x | **v2** |
| 编码性能 | ✅ 快 | ❌ 慢 1.8-3.4x | **v1** |
| 内存分配 | 中 | 解码优，编码差 | 看场景 |
| API 稳定性 | ✅ 稳定 | ⚠️ 实验 | **v1** |
| 错误处理 | 基础 | ✅ 详细 | **v2** |
| 灵活性 | 高 | ✅ 更高 | **v2** |

### 推荐策略

**现在（2026 Q1）**：
- 解码为主：试用 v2
- 编码为主：继续用 v1
- 性能关键：用 easyjson

**6 个月后**：
- v2 API 稳定后，逐步迁移解码路径
- 等待编码性能优化
